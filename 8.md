## **Topic 29: Joining & Blending Data**

### **Connecting to Multiple Data Sources**

Tableau can combine data from multiple tables/sources in two main ways:

1. **Joins** - Combine tables in data source (before analysis)
2. **Data Blending** - Combine data sources in worksheet (during analysis)

---

### **JOINS** ‚≠ê‚≠ê‚≠ê

**What is a Join?**

**Join** = Combining rows from two or more tables based on a related column

**Happens:** At data source level (before you build views)

**Result:** Single unified data table

---

### **Types of Joins**

Tableau supports 4 join types:

#### **1. INNER JOIN** (Most restrictive)

**Shows:** Only rows that exist in BOTH tables

**Venn Diagram:**
```
Table A    Table B
   ‚ï≠‚îÄ‚îÄ‚ïÆ‚ï≠‚îÄ‚îÄ‚ïÆ
   ‚îÇ  ‚îÇ‚îÇ  ‚îÇ
   ‚ï∞‚îÄ‚îÄ‚ïØ‚ï∞‚îÄ‚îÄ‚ïØ
     ‚ñì‚ñì  ‚Üê Only this area (overlap)
```

**Example:**
```
Orders (Table A)        Customers (Table B)
Order | Customer_ID     Customer_ID | Name
1     | 101            101         | Alice
2     | 102            102         | Bob
3     | 105            103         | Carol

INNER JOIN result:
Order | Customer_ID | Name
1     | 101        | Alice
2     | 102        | Bob
(Order 3 excluded - no matching customer)
```

**Use when:** You only want complete records (both tables have data)

---

#### **2. LEFT JOIN** (Most common)

**Shows:** ALL rows from left table, matching rows from right (NULLs if no match)

**Venn Diagram:**
```
Table A    Table B
   ‚ï≠‚îÄ‚îÄ‚ïÆ‚ï≠‚îÄ‚îÄ‚ïÆ
   ‚îÇ‚ñì‚ñì‚îÇ‚îÇ  ‚îÇ
   ‚ï∞‚îÄ‚îÄ‚ïØ‚ï∞‚îÄ‚îÄ‚ïØ
   ‚ñì‚ñì‚ñì‚ñì  ‚Üê All of A, matching B
```

**Example:**
```
Orders (LEFT)           Customers (RIGHT)
Order | Customer_ID     Customer_ID | Name
1     | 101            101         | Alice
2     | 102            102         | Bob
3     | 105            103         | Carol

LEFT JOIN result:
Order | Customer_ID | Name
1     | 101        | Alice
2     | 102        | Bob
3     | 105        | NULL
(Order 3 kept, Name is NULL)
```

**Use when:** Primary table is orders/transactions, want all records

---

#### **3. RIGHT JOIN**

**Shows:** ALL rows from right table, matching rows from left (NULLs if no match)

**Same as LEFT JOIN but reversed**

*Rarely used - just switch table order and use LEFT!*

---

#### **4. FULL OUTER JOIN**

**Shows:** ALL rows from both tables (NULLs where no match)

**Venn Diagram:**
```
Table A    Table B
   ‚ï≠‚îÄ‚îÄ‚ïÆ‚ï≠‚îÄ‚îÄ‚ïÆ
   ‚îÇ‚ñì‚ñì‚îÇ‚îÇ‚ñì‚ñì‚îÇ
   ‚ï∞‚îÄ‚îÄ‚ïØ‚ï∞‚îÄ‚îÄ‚ïØ
   ‚ñì‚ñì‚ñì‚ñì‚ñì‚ñì  ‚Üê Everything
```

**Example:**
```
Orders                  Customers
Order | Customer_ID     Customer_ID | Name
1     | 101            101         | Alice
2     | 102            102         | Bob
3     | 105            103         | Carol

FULL OUTER JOIN result:
Order | Customer_ID | Name
1     | 101        | Alice
2     | 102        | Bob
3     | 105        | NULL
NULL  | 103        | Carol
(All orders + all customers)
```

**Use when:** Need to see everything from both tables (rare)

### **Creating Joins in Tableau**

**Step-by-step:**

1. **Connect to first table**
   - Data Source tab
   - Drag table to canvas

2. **Add second table**
   - Drag second table next to first
   - Join icon appears

3. **Configure join**
   - Venn diagram appears
   - Click to choose join type
   - Define join clauses

4. **Join Clauses**
   - Which fields link tables
   - Can have multiple clauses
   - Operators: =, >, <, >=, <=, ‚â†

---

### **Join Clause Configuration**

**Simple join (single field):**
```
Orders.Customer_ID = Customers.Customer_ID
```

**Multiple clauses (AND logic):**
```
Orders.Customer_ID = Customers.Customer_ID
AND
Orders.Region = Customers.Region
```

**Custom operators:**
```
Orders.Order_Date >= Promotions.Start_Date
AND
Orders.Order_Date <= Promotions.End_Date
```
*Range join - advanced!*

### **Join Issues & Solutions** ‚≠ê **EXAM IMPORTANT!**

**Problem 1: Duplicate Rows**

**Scenario:** 
```
Orders: 1 row per order
Products: Multiple products per order
JOIN ‚Üí Multiplies rows!
```

**Example:**
```
Orders              Products
Order_ID | Amount   Order_ID | Product
1001     | 100     1001     | Chair
                   1001     | Desk

After JOIN:
Order_ID | Amount | Product
1001     | 100    | Chair
1001     | 100    | Desk

Now SUM(Amount) = 200 (WRONG! Should be 100)
```

**Solutions:**
- Use proper granularity fields
- Aggregate at correct level
- Use LOD expressions
- Consider blending instead

---

**Problem 2: NULLs from Outer Joins**

**Left join creates NULLs in right table fields**

**Handle with:**
```
IFNULL([Field], "Unknown")
ZN([Numeric_Field])
```

**Or filter:**
- Filter: `[Field] IS NOT NULL`

---

**Problem 3: Wrong Join Type**

**Symptoms:**
- Missing data (used INNER, should use LEFT)
- Too much data (used FULL OUTER unnecessarily)
- Duplicates (many-to-many join)

**Solution:** Review join type and relationships

---


**After join, field names prefixed with table name (if ambiguous)**

### **Cross-Database Joins** ‚≠ê

**Join tables from DIFFERENT databases!**

**Example:**
- Orders from SQL Server

- Products from MySQL

**Requirements:**
- Tableau Desktop (not Web)
- Local processing (extracts recommended)

**Performance:**
- Slower than single-database
- Create extract for better performance

### **Union vs Join**

**Don't confuse!**

**JOIN:**
- Combines COLUMNS (side by side)
- Based on matching keys
- Different tables, related data

**UNION:**
- Combines ROWS (stacked)
- Same structure
- Appending similar data

**Example:**
```
JOIN (combine columns):
Table A + Table B = More columns, matching rows

Orders     Customers    ‚Üí    Orders with Customer Names
Order_ID   Cust_ID          Order_ID | Cust_ID | Name | Amount
Amount     Name             

UNION (combine rows):
Jan_Sales + Feb_Sales = More rows, same columns

Jan_Sales     ‚Üí    Combined_Sales
Order_ID           Order_ID | Amount | Month
Amount            (All months stacked)

Feb_Sales
Order_ID
Amount
```

---

### **Creating Unions**

**Step-by-step:**

1. **Data Source tab**
2. **Drag first table to canvas**
3. **Drag second table BELOW first** (not beside)
4. **Union created automatically**

**Or:**

1. **Double-click "New Union"**
2. **Select tables to combine**
3. **Wildcard union** (e.g., all sheets matching pattern)

---

**Union Requirements:**
- Same number of columns (or Tableau pads)
- Similar data types
- Column names matched by position or name

**Union adds fields:**
- **Table Name** - Which table row came from
- **Sheet** - Which sheet (if Excel)

### **DATA BLENDING** ‚≠ê‚≠ê

**What is Data Blending?**

**Blending** = Combining data from separate data sources at visualization level

**Differences from Joins:**

| Feature | Join | Blend |
|---------|------|-------|
| **When** | Data source level | Worksheet level |
| **Result** | Single table | Separate sources |
| **Performance** | Faster | Slower |
| **Aggregation** | Row-level available | Pre-aggregated |
| **Use case** | Same database | Different sources |

---

### **When to Use Blending**

‚úÖ **Good for:**
- Different data sources (SQL + Excel)
- Different granularities
- Secondary data for comparison
- Small lookup tables
- Quick analysis without joins

‚ùå **Not ideal for:**
- Large datasets (slow)
- Row-level calculations needed
- Same database (use joins)
- Complex relationships

---

### **How Blending Works**

**Concept:**

1. **Primary Data Source** - Main data, all dimensions available
2. **Secondary Data Source** - Supplementary, linked dimensions only
3. **Linking Field** - Common dimension(s) connecting them

**Visual indicators:**
- **Primary:** Blue checkmark
- **Secondary:** Orange checkmark
- **Linked field:** Orange chain link icon

---

### **Creating a Blend**

**Step-by-step:**

1. **Connect to first data source** (Primary)
   - Build view with this data

2. **Connect to second data source** (Secondary)
   - Switch to this data source (dropdown)

3. **Add field from secondary to view**
   - Tableau auto-creates blend

4. **Verify/edit blend relationship**
   - Data ‚Üí Edit Relationships
   - Ensure linking fields correct

---

**Example:**

**Primary:** Sales Data (Order_ID, Product, Sales, Date)
**Secondary:** Targets (Product, Monthly_Target)

**Blend on:** Product

**View:** Product, SUM(Sales), SUM(Monthly_Target)

**Result:** Sales vs Target by Product

---

### **Blend Relationship**

**Linking fields must:**
- Same data type
- Matching values
- Similar granularity

**Auto-detected by:**
- Same field name
- Same role (Geographic, Date)

**Manual configuration:**
- Data ‚Üí Edit Relationships
- Select linking fields
- Can be multiple fields

---

### **Active Link Fields**

**Orange chain link icon** = Actively linking

**Click chain link to activate/deactivate:**
- Active: Blend uses this dimension
- Inactive: Dimension independent

**Use multiple links:**
```
Blend on Product AND Region
Both must match for data to combine
```

---

### **Blend Aggregation** ‚≠ê **IMPORTANT!**

**Key limitation:** Secondary source is **pre-aggregated**

**Example:**
```
Primary: Transaction-level (1 row per order)
Secondary: Already aggregated (1 row per product)

Can't access row-level detail from secondary!
```

**Implications:**
- Can't do row-level calculations
- Filtered differently than primary
- Asterisk (*) on secondary fields = aggregated

---

### **Blending Best Practices**

‚úÖ **DO:**
- Use for different data sources
- Keep secondary source simple
- Ensure clean linking fields
- Use extracts for performance

‚ùå **DON'T:**
- Blend when join possible (slower)
- Blend very large datasets
- Expect row-level detail from secondary
- Rely on blending for complex analysis

---

**How it works:**

### **Scenario-Based Questions:**

**Q1:** You have Orders (many) and Customers (one customer per order). Best join type to keep all orders?
- A) INNER JOIN
- B) LEFT JOIN (Orders left, Customers right) ‚úÖ
- C) RIGHT JOIN
- D) FULL OUTER

**Why?** LEFT JOIN keeps all orders (left table), adds customer info where available. INNER would drop orders without customer match.

---

**Q2:** After joining Orders and Order_Details, SUM(Sales) doubled. Why?
- A) Wrong join type
- B) Many-to-many join creating duplicates ‚úÖ
- C) Data error
- D) Wrong aggregation

**Why?** If Orders has multiple Order_Details per order, join multiplies Order rows, inflating sums. Solution: Use proper granularity or LOD.

---

**Q3:** You need to compare sales data (SQL) with budget data (Excel). Best approach?
- A) Import Excel to SQL, then join
- B) Cross-database join
- C) Data blend ‚úÖ (if simple comparison)
- D) Manually combine

**Why?** Blend is quick for comparing different sources, especially if budget is simple lookup. Cross-database join also works but more setup.

---

**Q4:** Secondary data source field shows asterisk (*). What does this mean?
- A) Calculated field
- B) Data quality issue
- C) Pre-aggregated in blend ‚úÖ
- D) Missing data

**Why?** Asterisk indicates field from secondary source in blend, already aggregated before combining with primary.

### **Pivot** ‚≠ê‚≠ê

**Wide format (bad for Tableau):**

1. **Data Source tab**
2. **Select columns to pivot** (Ctrl+Click multiple)
3. **Right-click ‚Üí Pivot**
4. **Creates two new columns:**
   - **Pivot Field Names** (Jan, Feb, Mar ‚Üí "Month")

5. **Rename columns as needed**

**Huge for cross-tab Excel data!**

### **Why Performance Matters**

**Slow dashboards = Frustrated users**

**Common complaints:**
- "Dashboard takes 2 minutes to load"

---

### **Extract vs Live Connection** ‚≠ê‚≠ê

**Live Connection:**
- Queries database every time
- Always current data
- Performance depends on database
- Network latency matters

**Extract:**
- Local copy of data
- Query from Tableau's engine
- Much faster (usually)
- Data refresh needed

---

**When to use Extract:**
‚úÖ Large datasets (millions of rows)
‚úÖ Slow database
‚úÖ Complex joins
‚úÖ Need speed
‚úÖ Offline work

**When to use Live:**
‚úÖ Data changes frequently (real-time)
‚úÖ Very small datasets
‚úÖ Fast database
‚úÖ Simple queries
‚úÖ Data security (no local copy)

---

### **Extract Optimization**

**Options when creating extract:**

**1. Filters:**
- Include only necessary data
- Date range (last 2 years)
- Specific regions
- Exclude test data

**2. Aggregation:**
- **Aggregate for visible dimensions**
- Rolls up to dimensions in workbook
- Dramatically reduces rows
- ‚ö†Ô∏è Can't drill to detail afterward

**3. Hide unused fields:**
- Excludes unnecessary columns
- Reduces size

**4. Sample:**
- Development only!
- Top N rows or random sample

---

**Extract Refresh:**
- **Full refresh:** Replace all data
- **Incremental refresh:** Add new/changed rows only (requires configuration)

**Schedule refreshes:**
- Tableau Server/Online
- Off-peak hours
- Balance freshness vs load

---

### **Query Optimization**

**Reduce query complexity:**

‚úÖ **Filter early:**
- Data source filters best
- Context filters next
- Reduce data volume

‚úÖ **Limit dimensions:**
- Only include needed fields
- Hide unused fields in data source

‚úÖ **Simplify joins:**
- Fewer joins = faster
- Inner joins faster than outer
- Relationships often better than joins

‚úÖ **Aggregate upstream:**
- Pre-aggregate in database
- Create summary tables
- Use database views

---

### **Calculation Optimization**


**Complex nested calcs:**
```

```

**LOD expressions:**
```
‚ö†Ô∏è Nested LOD very slow
‚úì Keep LOD simple

```

**Table calculations:**
```
‚úì Fast on small data
‚ö†Ô∏è Slow on large data (runs locally)
```

---

### **Dashboard Optimization** ‚≠ê

**Dashboard-specific tips:**

**1. Limit worksheet count:**
- 5-8 sheets optimal
- More sheets = more queries
- Combine where possible

**2. Filter with actions not filters:**
- Filter actions reduce queries
- Filter controls query for values

**3. Hide unused sheets:**
- Dashboard only loads visible
- Hide detail sheets until needed
- Show/Hide buttons helpful

**4. Optimize each worksheet:**
- Fast sheets = fast dashboard
- Fix slowest sheet first

**5. Use device layouts:**
- Mobile layouts can be simpler
- Fewer marks on small screens

---

### **Marks Optimization**

**Too many marks = slow rendering:**

**Guideline:**
- <1,000 marks: Fast
- 1,000-10,000: OK
- 10,000-100,000: Slow
- >100,000: Very slow

**Solutions:**

‚úÖ **Aggregate more:**
- Show months instead of days
- Show categories instead of products

‚úÖ **Filter:**
- Top N
- Date range
- Relevant subset

‚úÖ **Summary tiles instead of scatter:**
- Average per category (50 marks)
- Instead of all transactions (50,000 marks)

‚úÖ **Sampling:**
- For visualization only
- Random sample for patterns

---

### **Tips for Specific Chart Types**

**Scatter plots:**
